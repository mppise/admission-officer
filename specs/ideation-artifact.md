# Ideation Artifact — University Admission Officer

> **Note:** This artifact was generated by spec-gantry reverse-engineering from an existing codebase (v4.0.0 in production). Review and update as needed.

## Problem statement

High school students face overwhelming complexity in the college admissions process: researching universities, understanding fit, writing persuasive essays, and managing application deadlines. They lack personalized, AI-driven guidance tailored to their academic profile and university-specific requirements.

## Users

- **Primary:** High school students (grades 11–12) preparing college applications
- **Secondary:** School counselors seeking to supplement manual guidance with AI tools
- **Usage context:** Command-line interface, batch processing of student/university profiles

## Success criteria

- Students can build and manage a personal profile (academic stats, achievements, goals)
- Students can extract and store university profile data (requirements, essay prompts)
- AI generates personalized college-fit guidance based on student + university profiles
- AI assists with essay brainstorming and writing (helps overcome writer's block)
- Users can export comprehensive guidance reports to PDF for offline review
- All operations guided through a responsive, keyboard-navigable CLI

## Feasibility notes

- **Tech maturity:** High. TypeScript, Ink.js, and Google Generative AI are well-established, well-documented, and production-ready.
- **Dependencies:** Lightweight. Primary dependencies are Google AI SDK, Playwright/Puppeteer for scraping, and Ink.js for UI. All have stable maintenance.
- **Scope:** Manageable. 2,577 lines of TypeScript across 7 logical components, each with clear boundaries.
- **Market:** Growing. AI-assisted college guidance is a recognized need; comparable tools exist (e.g., Naviance, Tuition Fit) but often require subscriptions.

## Key decisions already made

1. **Language & Runtime:** TypeScript + Node.js (v20+) for cross-platform CLI
2. **UI Framework:** Ink.js (React for the terminal) for responsive, keyboard-friendly navigation
3. **AI Integration:** Google Generative AI API (not OpenAI) for LLM capabilities
4. **Data Persistence:** Local filesystem (user's workspace directory) for student/university profiles and generated content
5. **Configuration:** Environment-driven (API keys, model selection, token budgets) via `.env` and saved config
6. **Browser Automation:** Playwright + Puppeteer for scraping university websites and generating PDFs
7. **Architecture:** Component-based (c01–c08) with centralized prompt management and shared utilities
8. **Packaging & Distribution:** Published to npm as `ao` CLI tool with automatic browser dependency management

## Open questions

- **Test Coverage:** Current codebase has 0 test files. Should this be addressed? What testing strategy (unit, integration, e2e)?
- **University Data Scraping:** How are university websites reliably parsed as they change? Is a static database + manual updates preferred?
- **Essay Advisor Depth:** Should the essay advisor offer iterative feedback (multiple rounds)? Or one-shot generation?
- **Scalability:** If demand grows, should this move from local FS to a backend service + database?
- **Multi-language Support:** Currently English only. Are non-English markets in scope?
- **Model Selection:** Currently uses Google Generative AI. Should we support multiple LLM providers (OpenAI, Claude, Anthropic)?
