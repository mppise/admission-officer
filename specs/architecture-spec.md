# Architecture Spec — University Admission Officer

> **Note:** Generated by spec-gantry reverse-engineering from v4.0.0 codebase. Review and update as needed.

## Vision

Provide high school students with personalized, AI-driven guidance for college admissions through an intelligent CLI tool that integrates student profiles, university data, and advanced language models to generate tailored essays and fit assessments.

## System overview

**University Admission Officer (AO)** is a command-line application that orchestrates a workflow for college-bound students:

1. **Profile Management** — Students define their academic stats, achievements, and goals
2. **Data Gathering** — Extract university requirements and essay prompts from university websites
3. **AI-Driven Guidance** — LLM generates personalized college-fit assessments
4. **Essay Assistance** — LLM brainstorms and drafts essays for each university
5. **Export** — Compile guidance and drafts into a printable PDF report

The system is driven by a keyboard-navigable CLI menu (Ink.js) that manages state in the local filesystem and communicates with the Google Generative AI API for intelligent analysis.

## Guardrails

1. **Component Naming:** Features are prefixed c01–c08 (e.g., c02-student-profile) for namespace isolation and discoverability.

2. **Prompt Management:** All LLM prompts are stored as Markdown files under `src/ai/prompts/`. Prompts are versioned with code and loaded at runtime. Naming convention: `c{NN}-{feature}-{purpose}.prompt.md`.

3. **Configuration:** Sensitive data (API keys, model names, token budgets) are read from environment variables or a persisted config file. Bootstrap validates all required settings at startup.

4. **Workspace Isolation:** Each student's data is stored in a separate directory under the workspace. Data is never uploaded to external services—only prompts and student descriptions are sent to the LLM.

5. **TypeScript Strict Mode:** All source files use strict mode (`"strict": true` in tsconfig.json). No `any` types without explicit `// @ts-expect-error` comments.

6. **Error Handling:** Errors from external APIs (Google AI, Playwright) are caught and presented to the user as readable messages. Critical errors (missing workspace, API key invalid) halt execution with guidance.

7. **Browser Automation:** Playwright is preferred for scraping (modern, fast). Puppeteer is used only for PDF rendering. Both are downloaded during `npm install` via `postinstall` hook.

8. **Ink.js UI Patterns:** All menus are keyboard-navigable (arrow keys, Enter, Escape). No mouse input required. Async operations show spinners. Long-running tasks (LLM calls, scraping) run in spinners without blocking the UI.

9. **No External Database:** Data is stored locally in the user's workspace filesystem. Enables offline work and privacy, but scales only to single-machine usage.

10. **npm Publishing:** The tool is distributed as a published npm package (`ao`). The `bin` field points to the compiled CLI entry point. Browser dependencies are auto-installed via `postinstall` script.

## Feature decomposition

Seven features were identified from the codebase:

1. **FEATURE-001** — CLI Shell & Navigation (c01-cli-shell)
   - Main entry point, menu system, command dispatcher
   
2. **FEATURE-002** — Student Profile Manager (c02-student-profile)
   - Create, edit, display student profiles (academics, achievements, goals)
   
3. **FEATURE-003** — University Profile Extractor (c03-university-profile)
   - Scrape university websites, extract requirements and essay prompts
   
4. **FEATURE-004** — AI Guidance Engine (c04-guidance-engine)
   - Generate personalized college-fit assessments via LLM
   
5. **FEATURE-005** — Essay Writing Advisor (c05-essay-advisor)
   - Brainstorm and draft essays for each university
   
6. **FEATURE-006** — PDF Report Exporter (c06-pdf-exporter)
   - Compile guidance + essays into a printable PDF
   
7. **FEATURE-007** — Status Bar & Message Queue (c08-status-bar)
   - Display status messages, logs, and operation feedback in the TUI

## External dependencies

- **Google Generative AI API** — LLM inference for guidance and essays (requires valid API key, billable)
- **Playwright / Puppeteer** — Browser automation for web scraping and PDF rendering (downloads browsers automatically)
- **Ink.js** — Terminal UI framework (npm dependency)
- **Node.js 20+** — Runtime (user must have Node.js installed)

## Unknowns / items to verify

1. **Test Coverage:** Codebase has 0 tests. No unit, integration, or e2e tests are present. Recommend defining test strategy.
2. **University Data Freshness:** How is university data updated when websites change? Is scraping reliable, or should there be a manual curated database?
3. **Essay Advisor Iterations:** Should essays go through multiple rounds of feedback, or is one generation sufficient?
4. **Scalability Plan:** Current design (local FS, single-user CLI) cannot scale to a multi-user SaaS. Is that in scope?
5. **Documentation:** Current README is sparse. Suggest expanding with setup instructions, examples, and troubleshooting.
6. **Model Flexibility:** Currently locked to Google Generative AI. Should we support provider switching (OpenAI, Anthropic) in the future?
7. **Privacy Guarantees:** Should the tool encrypt or hash student data at rest? Current storage is plaintext in local FS.
8. **Accessibility:** Is the terminal UI accessible to screen readers? Are color indicators backed by text labels?
